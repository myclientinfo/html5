Introduction

What is HTML5?

HTML5 is a specification from the W3C intended to upgrade and improve areas of the web that were not performing ideally with a 12 year old standard.

It adds new behaviours, tags, and attributes that allow functionality that could not have been conceived at the time: rich web apps, multimedia, browsers that actually work.

What isn't HTML5

HTML 5 is not a collection of techniques and methodologies, not limited to:

  • Ajax
  • CSS3
  • Javascript
  • Responsive Design
  • Web X.0
  • A lot of other things

The difference between HTML 5 and HTML 4

The key change is that 5 intends to be do for real what HTML 4 was happy to fake. Tags in HTML 5 have more meaning than they used to.

Tags, documents, and data within documents is supposed to have more inherent meaning. More on this soon.

Browser support

By far the biggest concern or hesitation people have about implementing HTML 5 functionality within either current or upcoming development is that the browsers people use will not support HTML 5, and a large number of users will not gain the benefits, meaning a large amount of work will be wasted.

Browser support is not an issue. It's an excuse.

Every single browser supports HTML 5, including Internet Explorer 6.

For a certain value of "support".

HTML 5 is not an "all or nothing" choice. If certain tags don't work... don't use them. If a certain techniques aren't reliable for your target market, use something else. There is nothing to break, and basic web development understanding is all that is needed to start gaining benefit from HTML5 and its approach to the web.

How to HTML5

Doctype, language, link and script tags simplified

Most developers will be familiar with this as a doctype:

<!DOCTYPE html PUBLIC "-//W3C//DTD XHTML 1.0 Transitional//EN" "http://www.w3.org/TR/xhtml1/DTD/xhtml1-transitional.dtd">

HTML 5 does away with the unnecessary and arcane parts of that and goes with the following.

<!DOCTYPE html>

With that in place it is an HTML 5 document.

Simplified syntax

Parts of this were actually relevant.

<!DOCTYPE html PUBLIC "-//W3C//DTD XHTML 1.0 Transitional//EN" "http://www.w3.org/TR/xhtml1/DTD/xhtml1-transitional.dtd">

In particular the reference to XHTML told the browser that this is XML based document. In other words everything had to validate according to strictish XML rules. Most particularly this means all tags had to be closed, and all attributes a quoted key/value pair.

In XHTML the following would not be valid:

<input type="checkbox" name="myCheckBox" checked>

In HTML5 this is considered perfectly fine. It's no longer XML, so it doesn't matter. Tags like <img> and <link> are free to do what they do best.

Shims, shivs and hacks

Unfortunately, all is not necessarily perfect in HTML5 implementation. When I said all browsers support HTML 5, there are certain definitions of "support" that apply. One of the major benefits of the new tags and syntax is that browsers that don't support something will almost always simply render it as it always had. There is no downside, with the exception of, as always, early versions of Internet Explorer.

Versions below 8 have a "feature" where they don't properly support the new structural elements. And because they don't understand them they can't be styled with CSS.

Shims, shivs and hacks

Shims, shivs and hacks

These hacks include "shims" and "shivs". The difference between these things is purely spelling. Another term is "polyfills". They're intended to put back in functionality that might be missing from a specific browser or class of browsers. Fortunately updating IE6+ to support the new structure tags is a simple javascript file.

Semantics

What is a semantic?

Semantics literally is the study of meaning. The point of semantics in HTML is probably best explained by showing what semantics is not.

<span class="time">Sunday, December 9th, 7:34pm</span>

This is not a time. To a human it reads like a time. But there's actually nothing about it that makes it a time. The class could say anything, and it would be no more, or less, a time.

Just like in the below, there's nothing about this code that makes it actually a footer. It's just called that.

<div id="footer">......</div>

Why you should care?

This is the key question that enthusiasts and advocates often overlook. "Semantics" for their own sake are relatively meaningless.

Semantically correct markup is more:

  • Consumable
  • Accessible
  • Scriptable
  • Indexable
  • Flexible

But also you should care... because you should care.

New structural tags

HTML 5 adds a couple of major new structural tags to the soup that is HTML. Among them are header, footer, nav, and section.

While the first three are fairly obvious, section may need some explaining.

A good guide to when to use a "section" is that if you wrote up a table of contents, would it have a listing. Note that to be valid a section must have a heading. Do not use section as a replacement for div.

DON'T USE TABLES!

DON'T USE TABLES!

Using tables for layout is the opposite of semantic. It creates extremely verbose HTML that is harder to navigate in the DOM as the code has no inherent meaning.

There is no good case for using tables for positioning and layout in today's browser environment. There are better methods, especially using CSS. That's what it's for.

USE TABLES!

Some people take it too far. The point is not to ban tables, but to use them for what they're for. If you have rows and columns of data, that's exactly what a table is intended for!

data-* attributes

HTML5 allows the creation of completely valid attributes to any element, as long as the attribute starts with "data-". This means data can be stored directly on the element, rather than needing to clutter javascript with global variables, or a form with hidden values.

<table id="products" data-sortcol="column3" data-sortdirection="asc" >

Conveniently, jquery provides excellent ways to address that information in its data() method.

$('#products').data('sortcol');

This sort of concept can even store entire arrays and structures.

<table id="products" data-sorting="{"sortby":"column3", "direction":"asc"}">

Form Fields

Tags: Progress

Progress fields do exactly what they sound like. They create a progress bar. The one at the top in this example is "indeterminate". The other one is nearly done. The value of progress can be easily scripted, allowing it to increase visibly.

Type: Email

The email type field provides little visible difference to a user, but provides mobile users a more customised keyboard to optimise text input. Additionally, the email field will self-validate.

Note that there is absolutely no scripting for this behaviour. It is all internal, default behaviour.

<input name="email" type="email" placeholder="email">

Type: All the others






There are a range of others of various degrees of usefulness and implementation.

  • color: Gives the end user a native color picker to choose a color.
  • date: Offers a datepicker.
  • datetime: An element to choose both date and time.
  • email: A field for entering e-mail address(es).
  • number: Picking a number.
  • tel: Choosing a telephone number.
$('input[type=date]').jDatePicker();

Attribute: required

A useful new attribute is "required". This has long been a requirement, a more syntactic and reasonable way for the field to be forced to be filled. Note that this doesn't actually replace the use of a * or something as a visual cue.

Attribute: placeholder

As seen in the previous examples, "placeholder" allows a non-filled field to have a label attached. This placeholder will then vanish when the user either clicks on the field, or starts entering data (depending on the browser).

Attribute: pattern

Allow the developer to specify a regular expression pattern that has to be matched by the field, for example, allowing it to be forced to numbers only, or a specific length.

It should be noted that it's important that the developer make the pattern apparent to the user, not like this example.

<input name="postcode" type="text" placeholder="postal code" pattern="[0-9]{4}">

Attribute: list

As seen in the previous examples, "placeholder" allows a non-filled field to have a label attached. This placeholder will then vanish when the user either clicks on the field, or starts entering data (depending on the browser).

Attribute: autofocus

Immediately give this field the focus, so the user can start typing straight away. Note: this can interact poorly with placeholder.

Drag and drop

Though it's a bit complex for this example, HTML5 supports "drag" and "drop" as events, with a range of elements being drag and droppable. In particular the API supports dragging and dropping files from the file system, and has significantly improved file upload capabilities over HTML4.
